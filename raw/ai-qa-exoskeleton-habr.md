# AI-экзоскелет для QA-инженера: от идеи до 11 автономных агентов

## Проблема

QA "отстаёт" от разработки:
- 80% времени уходит на рутину
- Автотесты не успевают за сценариями
- Рост бюджета на QA не спасает

### Распределение времени QA

| Активность | % времени |
|------------|----------|
| Анализ требований и тест-дизайн | 20% |
| Создание тест-кейсов в TMS | 15% |
| Подготовка тестовых данных | 10% |
| Автоматизация тестовых сценариев | 25% |
| Проведение тестирования | 15% |
| Регресс | 10% |
| Подготовка отчётов | 5% |

## Идея: QA как оператор AI-ассистента

**Принцип:** QA-инженер не заменяется, а усиливается.

| Было | Стало |
|------|-------|
| Инженер лично выполняет каждый этап | Инженер ставит задачи ассистенту |
| Каждый использует свои промпты | Единые правила работы |
| Ручное управление данными | AI генерирует в привычных форматах |
| Результат без контроля | Контроль на каждом этапе |

## Архитектура: 11 скиллов

### Pipeline
```
Jira/Confluence → Загрузка → Декомпозиция → Сценарии → Автоматизация → Валидация → Доставка
                          требований      требований    (API+UI)      и доставка   (GitLab, Zephyr)
```

### 11 скиллов

| # | Скилл | Вход | Выход |
|---|-------|------|-------|
| 1 | **jira-fetch** | URL задачи | Markdown + вложения |
| 2 | **requirements-decomposer** | Markdown | Структурированные US/Task |
| 3 | **scenarios-generator** | Требования + Figma | JSON-сценарии + RTM |
| 4 | **scenario-data-fix** | Сценарии + Global Context | Обновлённые сценарии |
| 5 | **scenario-discrepancies-fix** | ERRORS_DISCREPANCIES.md | Исправленные сценарии |
| 6 | **selectors-collector** | Сценарии | SELECTORS.json (POM) |
| 7 | **automation-api** | JSON-сценарии | pytest + requests |
| 8 | **automation-ui** | JSON + SELECTORS | pytest + Playwright |
| 9 | **automation-compare** | JSON + pytest | COMPARISON_REPORT.md |
| 10 | **zephyr-uploader** | JSON-сценарии | Кейсы в TMS |
| 11 | **mr-creator** | Автотесты | MR в GitLab |

## Результаты за 1.5 месяца пилота

| Метрика | До | После |
|---------|-----|-------|
| Регрессионных тест-кейсов | ~50 | ~400 |
| Автоматизация | Отсутствовала | ~100% покрытие |
| Время регресса | ~1 день | Десятки минут |
| Время от завершения до апрува QA | Несколько дней | Несколько часов |
| Подключение нового проекта | Месяцы | ~4 часа |

## Ключевые решения

### 1. JSON-сценарии как единый источник правды
```
Требования → JSON-сценарии → pytest-код
                   ↑                 │
                   └─── Валидация ───┘
```

### 2. Мутационное тестирование
```python
# Оригинал
assert response.status_code == 200
# Мутация
assert response.status_code == 400  # Должен упасть!
```
Если мутированный тест проходит → оригинальный тест ничего не проверяет.

### 3. Global Context
Централизованное хранилище знаний:
- API-контракты
- Тестовые данные
- Архитектура
- OpenAPI спецификации

## Грабли и решения

| Проблема | Решение |
|----------|---------|
| Контекстное окно LLM переполняется | Декомпозиция по горизонтали (US) и вертикали (этапы) |
| Галлюцинации при генерации | Global Context + мутационное тестирование |
| Конфликты параллельных агентов | Изоляция workspace + checkpoint-валидация |
| Ограничения платформы | Архитектурные обходы |

## Стоимость

Хватает на 2-3 проекта, ~100+ тестов/месяц/проект

## Чему научились

1. **Модульность** — каждый скилл независим и заменяем
2. **Global Context** — устраняет галлюцинации
3. **Качество > скорости** — checkpoint-валидация предотвращает каскад ошибок
4. **Авто + ручной контроль** — флаг `--auto` закрывает очевидное
5. **Инвестируйте в сбор контекста** — качество результата = качество входных данных
