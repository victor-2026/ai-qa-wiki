## Суть RTM

**RTM (Requirements Traceability Matrix)** — таблица, устанавливающая связи между требованиями, тест-кейсами и дефектами.

### Три вопроса, на которые отвечает RTM

| Вопрос | Ответ RTM |
|--------|-----------|
| Что мы протестировали? | Требования с тест-кейсами |
| Что мы НЕ протестировали? | Пробелы в покрытии |
| Почему мы тестируем именно это? | Связь тест → требование |

## Типы трассируемости

| Тип | Связь | Назначение |
|-----|-------|------------|
| Прямая (Forward) | Требование → Тест-кейс | Проверка покрытия |
| Обратная (Backward) | Тест-кейс → Требование | Зачем нужен тест |
| Двунаправленная | Требование ⇄ Тест-кейс | Полная трассировка |
| Вертикальная | Требование → Дизайн → Код → Тест | Сквозная трассировка |
| Горизонтальная | Тест → Баг → Исправление → Ретест | Трассировка внутри этапа |

## Структура RTM

| Колонка | Описание | Пример |
|---------|----------|--------|
| Requirement ID | Уникальный ID требования | REQ-AUTH-01 |
| Requirement Description | Краткое описание | Авторизация по email |
| Test Case ID | Тест-кейсы | TC-LOGIN-01, TC-LOGIN-02 |
| Test Status | Pass / Fail / Blocked / Not Run | Pass |
| Defect ID | Связанные баги | BUG-101 |
| Defect Status | Open / Fixed / Closed | Fixed |
| Coverage Status | Covered / Not Covered / Partial | Covered |
| Coverage % | Процент покрытия | 100% |

## Пример RTM

| REQ ID | Требование | Приоритет | Test Case ID | Статус | Defect ID | % покрытия |
|--------|------------|-----------|--------------|--------|-----------|------------|
| REQ-01 | Авторизация | High | TC-LOGIN-01, TC-LOGIN-02, TC-LOGIN-03 | Pass, Pass, Fail | BUG-101 | 100% |
| REQ-02 | Восстановление пароля | Medium | TC-REC-01, TC-REC-02 | Pass, Not Run | — | 50% |
| REQ-03 | Поиск товаров | High | TC-SEARCH-01, TC-SEARCH-02 | Pass, Pass | — | 100% |
| REQ-04 | Фильтрация по цене | Low | — | — | — | **0% (пробел!)** |

## Анализ влияния изменений

```
ИЗМЕНИЛИ REQ-01 (Авторизация)
                    │
                    ▼
┌─────────────────────────────────────────────────┐
│  RTM показывает:                                │
│  • TC-LOGIN-01 (Pass)     → Перезапустить      │
│  • TC-LOGIN-02 (Pass)     → Перезапустить      │
│  • TC-LOGIN-03 (Fail)     → BUG-101 требует    │
│                              ретеста            │
│                                                 │
│  REQ-01 также влияет на REQ-02                 │
└─────────────────────────────────────────────────┘
```

## Статистика покрытия

| Показатель | Значение |
|------------|----------|
| Всего требований | 50 |
| С тестами | 45 |
| БЕЗ тестов | 5 |
| **Процент покрытия** | **90%** |
| С дефектами | 12 |
| Тест-кейсов всего | 180 |

## Типичные ошибки

| Ошибка | Последствие |
|--------|-------------|
| RTM не обновляется | Быстро устаревает |
| Формальная привязка | Ложное ощущение покрытия |
| Нет связи с дефектами | Сложно анализировать проблемы |
| Дублирование тестов | Искажает метрики |
| Нет обратной трассировки | "Висящие" тесты без назначения |

## Чек-лист QA по RTM

- [ ] Каждое требование имеет хотя бы 1 тест-кейс?
- [ ] Есть "висящие" тест-кейсы без привязки к требованию?
- [ ] При изменении требования обновлены связанные тесты?
- [ ] Каждый дефект привязан к конкретному требованию?
- [ ] Можно ответить: "Почему этот тест не пройден?" через RTM?
- [ ] RTM обновляется после каждого релиза/спринта?

## Инструменты

| Инструмент | Тип | Особенность |
|------------|-----|-------------|
| Jira + Xray/Zephyr | Плагин | Встроенная трассировка |
| TestRail | TMS | Нативная поддержка RTM |
| Qase | TMS | Простая связь требований |
| Excel/Google Sheets | Ручной | Бесплатно, но трудоёмко |
| Polarion | ALM | Enterprise решение |
| Azure DevOps | ALM | Встроенная трассировка |

## Когда RTM НЕ нужна

| Ситуация | Альтернатива |
|----------|--------------|
| Стартап с 5-10 требованиями | Trello / Notion |
| Проект на 1-2 недели | RTM не окупится |
| Команда из 1-2 человек | Вся трассировка в голове |
| Ad-hoc тестирование | Нет чётких требований |
| Прототип / MVP | Требования меняются каждый день |

### Правило

> **>20 требований + команда от 3 человек = RTM окупается**

## Чек-лист перед внедрением

- [ ] В проекте больше 20 требований?
- [ ] Команда больше 3 человек?
- [ ] Требования меняются не каждый день?
- [ ] Есть кто-то, кто будет обновлять матрицу?
- [ ] Готовы тратить 15-30 минут в спринт на поддержку?

**3+ ответов "да" → внедряйте RTM**

---

## Ключевые выводы

1. **RTM — инструмент, не самоцель.** Ценность в ответах, которые она даёт.
2. **Начните с малого.** Простая таблица: Требование → Тест-кейс.
3. **Главное — актуальность.** Мёртвая матрица хуже её отсутствия.
4. **>20 требований + команда 3+ = RTM сэкономит нервы.**

> **RTM — это не бюрократия, а навигатор тестировщика.**
